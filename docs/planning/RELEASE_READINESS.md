## M0 readiness gate

The M0 implementation is verified locally. Governance, ADRs, health slice, lockfiles, log-redaction checks, local format/lint/typecheck/test/build, isolated fresh-clone setup, CLI execution, debug app bundle, and live IPC UI smoke have passed. M0 itself remains **in progress** until a GitHub remote exists and the macOS/Windows/Linux matrix is green.

## M1 local readiness note

The resumable scanner is safe to exercise with test folders: progress and pending paths persist in SQLite, live manifest rows are not replaced until a complete scan publishes atomically, pause is acknowledged between entries, expired work reopens as interrupted, and resume revalidates the original canonical authorization boundary. This is not yet a release claim: Windows junction/hidden-attribute runtime evidence, peak RSS, remote CI, and live UI interaction evidence remain open.

## M2 extraction local readiness note

The text/Markdown/code, text-layer PDF, DOCX/PPTX/XLSX, image-metadata, and macOS PNG/JPEG Screenshot OCR slices are safe to exercise with test files already present in an explicit scanned scope. Providers never receive an arbitrary path; source size and actual open-handle identity are revalidated; reads/decompression/pages/archive/XML/chunks/image probe/dimensions/OCR pixels/output/observations/caller time are bounded. PDF active content/attachments and Office macros/formulas/relationships/embeddings remain inert; all text including OCR stays `untrusted_extracted_text`; image metadata decodes no pixels or EXIF/GPS and publishes separately from FTS. OCR stores mandatory spatial and optional honest confidence provenance and publishes atomically only after complete provider/source validation. CLI job status, image metadata, and Desktop dashboard payloads contain no paths or text. The Desktop may explicitly start, cancel, retry, or resume OCR from an eligible already-scanned Search result; IPC accepts only scope/node/job IDs, rejects generic extraction jobs, and delegates media, scope and identity authorization to the Rust core.

The Windows provider is code-ready, not runtime-ready. It is package-identity gated, validates the resolved recognizer after requesting Traditional Chinese and English, accepts only absent/zero `TextAngle` for source-aligned boxes, and allows at most one owned cleanup worker if native cancellation outlives the caller deadline. Host state-machine/policy tests and Windows cfg check/Clippy do not prove WinRT execution. This is not M2 completion or a safe-to-exercise Windows claim: real Windows/MSIX identity, recognizer present/missing behavior, mixed/no-text/corrupt/limit OCR, native cancel/cleanup/single-worker recovery, RSS, packaged fallback, scanned-PDF OCR, representative corpora, macOS Intel/Linux runtime, live Desktop interaction, and 8 GB extraction benchmarks remain open.

## M3 lexical local readiness note

The FTS5 baseline is safe to exercise on test scopes: search is read-only, stays in bundled SQLite, accepts bounded quoted queries of at least three Unicode characters, caps candidates/results/snippets, excludes absent locations and stale chunks, and exposes fixed ranking explanations. Scope, match-source, extension, and modified-time filters are validated twice and echoed after normalization. User-invoked CLI/Desktop results intentionally return authorized paths and bounded untrusted snippets; ordinary logs omit query/path/text. A reproducible 10k synthetic macOS arm64 baseline records p50/p95/max and FTS bytes. This is not an M3 or release claim: graph-backed project/folder filters, vector/embedding providers, hybrid/semantic behavior, representative/100k/8 GB/RSS/thermal evaluation, short-query strategy, live Desktop interaction, and remote platform runtime remain open.

## M4 project-graph local readiness note

The Folder Profile and exact-root correction slices are safe to exercise on a test folder after a completed scan: they use only current explicit-scope locations, immutable evidence and append-only user decisions, with no model, membership edge or file action. The exact-duplicate slice is safe only as an explicit development check for two canonical current non-empty files in the same scope: it denies symlink aliases and same identities, caps each source at 64 MiB, compares every byte through read-only handles, and revalidates identity/metadata after reading. An exact-pair decision repeats that verification before appending immutable feedback. The filename-version slice is also explicit-only: both files pass the same scope/open-handle checks twice, names must share an NFC/lowercased base and extension with allowlisted numeric `vN` suffixes, and no file content or timestamp orders the versions. A version decision repeats live verification and applies only to equivalent directional evidence; changed direction returns to `suggested`. Explicit relation responses intentionally include current paths; structured logs and history omit paths, names, database location and content. This is not M4 or release completion: file membership, entities/topics, related/similarity and general version discovery, background duplicate discovery, larger-file hashing, cross-pair learning, merge/split, retrieval integration, Project UI, latency/RSS/8 GB and cross-platform evidence remain open.

## M6 watch-core local readiness note

The durable reconciliation core, native hint source and Desktop periodic fallback are safe to exercise on test scopes after an explicit completed Initial Manifest Scan: hints remain untrusted, scope/path/symlink policy is revalidated, temporary downloads are excluded by both hint and scanner paths, a stable read-only identity snapshot is required for ordinary hints, and only bounded batches of the existing atomic scanner can publish live metadata. Authorization alone never scans. One bounded non-blocking source routes native hints outside the callback; watch-set changes, overflow, rescan, source failure and unmatched paths request all-scope reconciliation. A second distinct path in one scope requests per-scope recovery, preventing an ordered temporary→final rename from being reduced to only the old path. An explicit root-only immediate transaction rechecks the event scope, exact stored authorization and completed scan before closing registration/recovery gaps and enforcing a five-minute maximum coalescing age without authorizing content extraction or a filesystem action; normal events retain their debounce. A recovery request received during an existing multi-batch scan remains pending and forces a fresh root scan after the old snapshot finishes. Direct ignored observations only update/create their aggregate and cannot cancel unrelated stabilizing work; only an explicit identified transition may remove its transferred nonterminal row, never user data or action history. The Tauri process resumes active events, keeps the five-minute safety path enabled, retains path-free degraded state, and bounds shutdown waiting without claiming graceful termination after timeout. macOS arm64 live source and Desktop create/modify/rename/delete/identity previously passed outside the restricted sandbox, including direct manifest metadata verification after modify. Deterministic ordered temp→final routing and manifest tests pass; the expanded live temp→final host run is pending after external execution quota denial. SQLite-level cancellation contention remains unbenchmarked. Runtime/event status is path-free and localized, and the latest packaged macOS arm64 accessibility tree verifies the four locale choices plus current Watch wording. This is not efficient incremental Watch Mode or M6 completion: create/delete/rename/subtree delta, incremental extraction/indexing, placeholder detection, pause/battery/thermal/low-memory policy, notifications, Smart Inbox, Windows/Linux/macOS Intel runtime, real storms/root replacement, large-tree/8 GB benchmark, tray/autostart behavior, and packaged live scan/search/OCR/rename workflows remain open.

## M5 rename-preview local readiness note

The bounded rename preview is safe to exercise on test files after a completed scan: it performs no filesystem mutation, denies symlink/reparse and path-fallback sources, matches manifest metadata and platform/open-handle identity, validates a portable same-folder name, fails on occupied destinations, and commits an immutable plan plus first append-only event atomically. Explicit CLI/Desktop preview returns paths and passed checks; logs and list/history summaries do not. The Desktop explicitly offers no execute control. This is not M5 completion or a usable organizer: Move, folder actions, execution, source revalidation immediately before action, destination verification/hash, cross-volume handling, process-kill/permission/disconnect fault injection, recovery, rollback, idempotent Undo, execution/recovery/Undo UI, live interaction, and Windows runtime evidence remain open.
