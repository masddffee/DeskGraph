## M9a native scope access local readiness note

The local Desktop implementation now owns folder selection in narrow Rust IPC rather than accepting a
frontend-supplied authorization path. On macOS, the local implementation creates an opaque,
versioned security-scoped bookmark, resolves it with explicit start/stop lifetime ownership,
refreshes a stale bookmark before reuse, and stores the scope and grant atomically. Migration
0020 deliberately treats every legacy scope without a grant, every corrupt grant, and every
unrestorable bookmark as `needs_reauthorization`; it never fabricates active access. The main
window has no frontend filesystem or dialog permission, and normal IPC/log payloads omit grant
bytes.

This is local implementation evidence only. The checked-in profile grants App Sandbox,
user-selected read/write, app-scoped bookmarks, and outbound client sockets. The last capability
is required for the macOS WebKit networking subprocess on the current arm64 host: an otherwise
identical ad-hoc sandbox copy logged a fixed network-permission failure and rendered blank, while
adding only `com.apple.security.network.client` restored the bundled UI and native picker. It does
not make a network connection a product requirement or authorize uploads; the current runtime has
no remote content client and the production CSP excludes the development HTTP/WebSocket origins.
`Entitlements.plist`, a bundle configuration, or an ad-hoc signature do not prove a production
sandbox. The signed/notarized clean-machine matrix
must prove first selection, restart restoration, stale refresh, user revocation, denial outside
the selected scope, legacy/corrupt reauthorization, and the accepted protected-container hostile
replacement probe. It must also capture outbound-connection attempts and prove that no filename,
path, content, OCR, embedding, or graph data leaves the device during offline core workflows. The
current non-macOS release branch intentionally fails closed rather than
claiming that a picker receipt is Windows package identity. Linux remains experimental; Windows
requires the full MSIX/package-family/update evidence before M9a or the Windows OCR/action fence
can be considered ready.

## M2 extraction local readiness note

The text/Markdown/code, text-layer PDF, DOCX/PPTX/XLSX, image-metadata, and macOS PNG/JPEG Screenshot OCR slices are safe to exercise with test files already present in an explicit scanned scope. Providers never receive an arbitrary path; source size and actual open-handle identity are revalidated; reads/decompression/pages/archive/XML/chunks/image probe/dimensions/OCR pixels/output/observations/caller time are bounded. PDF active content/attachments and Office macros/formulas/relationships/embeddings remain inert; all text including OCR stays `untrusted_extracted_text`; image metadata decodes no pixels or EXIF/GPS and publishes separately from FTS. OCR stores mandatory spatial and optional honest confidence provenance and publishes atomically only after complete provider/source validation. CLI job status, image metadata, and Desktop dashboard payloads contain no paths or text. The Desktop may explicitly start, cancel, retry, or resume OCR from an eligible already-scanned Search result; IPC accepts only scope/node/job IDs, rejects generic extraction jobs, and delegates media, scope and identity authorization to the Rust core.

The Windows provider is code-ready, not runtime-ready. It is package-identity gated, validates the resolved recognizer after requesting Traditional Chinese and English, accepts only absent/zero `TextAngle` for source-aligned boxes, and allows at most one owned cleanup worker if native cancellation outlives the caller deadline. Host state-machine/policy tests and Windows cfg check/Clippy do not prove WinRT execution. This is not M2 completion or a safe-to-exercise Windows claim: real Windows/MSIX identity, recognizer present/missing behavior, mixed/no-text/corrupt/limit OCR, native cancel/cleanup/single-worker recovery, RSS, packaged fallback, scanned-PDF OCR, representative corpora, macOS Intel/Linux runtime, live Desktop interaction, and 8 GB extraction benchmarks remain open.

## M3 lexical local readiness note

The FTS5 baseline is safe to exercise on test scopes: search is read-only, stays in bundled SQLite, accepts bounded quoted queries of at least three Unicode characters, caps candidates/results/snippets, excludes absent locations and stale chunks, and exposes fixed ranking explanations. ## M4 project-graph local readiness note

The Folder Profile and exact-root correction slices are safe to exercise on a test folder after a completed scan: they use only current explicit-scope locations, immutable evidence and append-only user decisions, with no model, membership edge or file action. The exact-duplicate slice is safe only as an explicit development check for two canonical current non-empty files in the same scope: it denies symlink aliases and same identities, caps each source at 64 MiB, compares every byte through read-only handles, and revalidates identity/metadata after reading. An exact-pair decision repeats that verification before appending immutable feedback. The filename-version slice is also explicit-only: both files pass the same scope/open-handle checks twice, names must share an NFC/lowercased base and extension with allowlisted numeric `vN` suffixes, and no file content or timestamp orders the versions. A version decision repeats live verification and applies only to equivalent directional evidence; changed direction returns to `suggested`. The screenshot-review slice is explicit-only and model-free: it groups current same-scope images only when active image metadata and completed screenshot-OCR provenance match, uses exact dimensions plus a bounded ten-minute window, reads no OCR text/pixels/names, and treats every member as independently reviewable. It does not claim screenshot origin, similarity, keeper, disposability or cleanup authorization. Explicit relation/group responses intentionally include current paths; structured logs and history omit paths, names, OCR text, database location and content. This is not M4, Smart Cleanup or release completion: file membership, entities/topics, related/similarity and general version discovery, broader screenshot discovery/correction, background duplicate/Inbox discovery, larger-file hashing, cross-pair learning, merge/split, retrieval integration, Project/Inbox UI, Trash/Undo, latency/RSS/8 GB and cross-platform evidence remain open.

ADR-029's Smart Cleanup Inbox read slice is safe to exercise on a test scope only after a completed scan and active access grant. Refresh is explicit, bounded to 20 recent sources and path-free: duplicate and version evidence live-reverifies before inclusion, screenshot groups bind current membership to an immutable observation, stale sources are omitted, and an incomplete bound/read/timeout stays visibly partial. Relation acceptance or rejection is graph correction and neither state enters the Cleanup Inbox as consent. CLI, Tauri and four-language Desktop output expose no path, filename, OCR, keeper, selection, reclaimable-space claim or file-action capability. This does not make Smart Cleanup release-ready: background discovery, correction, pagination/larger refresh policy, immutable independent selection/keeper/confirmation, D-017 platform Trash, per-item journal/receipt, crash recovery, Undo, cross-platform packaged runtime, accessibility E2E and 8 GB evidence remain open.

ADR-030's Cleanup durable Preview core and ADR-031's explicit local review boundary are locally verified. The independent immutable plan/journal binds one current suggested observation to a target and any required keeper, permits only older-version target/newer-version keeper, stores both sides' strong identity/hash/root/parent topology, and requires exact duplicates to retain matching final hashes. A user must explicitly open the current authorized source detail before selecting a permitted target/keeper pairing. Paths exist only in that transient response; the durable Preview and ordinary history/logs/preferences remain path-free. Tauri/Desktop expose only Review and Create Preview, and the result cannot authorize or execute an action. D-017 official-source review found conditional receipt candidates—macOS `trashItem` plus a returned-URL security-scoped bookmark, and Windows `ITransferSource::RecycleItem` plus a reconstructed PIDL/`TSF_NORMAL` move—but no documented exact held-source-bound Trash primitive on either platform and no unconditional Windows restart-durable receipt guarantee. Existing previews are therefore permanently non-confirmable. This closes review and evidence-binding only; a new immutable original-destination/receipt binding, evidence-bound confirmation, accepted platform action/fence, per-item execution journal, crash recovery, Undo, Windows runtime and packaged fault matrices remain release blockers.

## M6 watch-core local readiness note

The durable reconciliation core, native hint source and Desktop periodic fallback are safe to exercise on test scopes after an explicit completed Initial Manifest Scan: hints remain untrusted, scope/path/symlink policy is revalidated, temporary downloads are excluded by both hint and scanner paths, a stable read-only identity snapshot is required for ordinary hints, and only bounded batches of the existing atomic scanner can publish live metadata. Authorization alone never scans. One bounded non-blocking source routes native hints outside the callback; watch-set changes, overflow, rescan, source failure and unmatched paths request all-scope reconciliation. A second distinct path in one scope requests per-scope recovery, preventing an ordered temporary→final rename from being reduced to only the old path. An explicit root-only immediate transaction rechecks the event scope, exact stored authorization and completed scan before closing registration/recovery gaps and enforcing a five-minute maximum coalescing age without authorizing content extraction or a filesystem action; normal events retain their debounce. A recovery request received during an existing multi-batch scan remains pending and forces a fresh root scan after the old snapshot finishes. Direct ignored observations only update/create their aggregate and cannot cancel unrelated stabilizing work; only an explicit identified transition may remove its transferred nonterminal row, never user data or action history. The Tauri process resumes active events, keeps the five-minute safety path enabled, retains path-free degraded state, and bounds shutdown waiting without claiming graceful termination after timeout. macOS arm64 live source and Desktop create/modify/rename/delete/identity previously passed outside the restricted sandbox, including direct manifest metadata verification after modify. Deterministic ordered temp→final routing and manifest tests pass; the expanded live temp→final host run is pending after external execution quota denial. SQLite-level cancellation contention remains unbenchmarked. Runtime/event status is path-free and localized, and the latest packaged macOS arm64 accessibility tree verifies the four locale choices plus current Watch wording. This is not efficient incremental Watch Mode or M6 completion: create/delete/rename/subtree delta, incremental extraction/indexing, placeholder detection, pause/battery/thermal/low-memory policy, notifications, Smart Inbox, Windows/Linux/macOS Intel runtime, real storms/root replacement, large-tree/8 GB benchmark, tray/autostart behavior, and packaged live scan/search/OCR/rename workflows remain open.

## M5 rename-preview local readiness note

The bounded rename preview is safe to exercise on test files after a completed scan: it performs no filesystem mutation, denies symlink/reparse and path-fallback sources, matches manifest metadata and platform/open-handle identity, validates a portable same-folder name, fails on occupied destinations, and commits an immutable plan, bounded SHA-256/root/parent/source binding and first append-only event atomically. Explicit CLI/Desktop preview returns paths and passed checks; logs and list/history summaries do not. The internal closed command/recovery protocol and test-only no-overwrite prototype are development evidence only. ADR-026 resolves D-018 by keeping general Unix Rename/Move Preview-only; the deterministic counterexample reproduces the source-leaf race. ADR-027 resolves D-019's identity-first topology but deliberately leaves runtime unavailable; Windows needs its packaged private namespace, while macOS additionally needs supported-version protected-container replacement proof before its `flock` candidate can be accepted. Every production adapter still fails before database/journal/mutation side effects and CLI/Desktop help exposes no Execute, recovery, or Undo control. This is not M5 completion or a usable executor: packaged identity, accepted macOS/Windows process fences, Windows exact-handle and platform fault evidence, Move planning, System Trash, idempotent user-facing recovery/Undo, execution UI and live interaction remain open.

## M8 Desktop UI local readiness note

The six-view Desktop shell is safe for bounded navigation review in the real macOS arm64 Tauri application. Home, Search, Projects, Inbox, History, and Settings route only to currently implemented local capabilities; their four-language descriptions explicitly disclose incomplete Project Discovery, suggestion-only cleanup, and preview-only organization. The shell adds responsive navigation, visible keyboard focus, a working skip target, reduced-motion handling, and persistent local language selection without a dependency or network surface.

This is not full product acceptance or demo readiness. The current automated gate covers formatting, lint, TypeScript, 49 Vitest contracts, and the production web build; the latest app bundle and accessibility tree prove native rendering and basic navigation only. Full synthetic-scope authorization/scan/search/extraction/OCR/watch/restart/action recovery flows, automated Desktop E2E, screen-reader and cross-platform runtime matrices remain open. Product recording is intentionally deferred until those flows close; it must use synthetic data outside private user folders, stop at capabilities actually implemented, avoid notifications and sensitive paths, and undergo visual review before any GIF or video enters release assets.
