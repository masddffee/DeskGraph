# ADR-024: Native-first OCR providers with a packaged in-process fallback

- Status: Accepted
- Date: 2026-07-17

## Context

DeskGraph must extract Traditional Chinese and English screenshot text while remaining local-first and useful without Python, Docker, an API key, a downloaded LLM, or a separately installed OCR service. ADR-012 requires OCR to use a separate provider boundary that receives only core-controlled bounded image data. OCR output is untrusted, must retain honest spatial provenance, and may replace prior active OCR text only after complete success.

The provider choice must also cover macOS arm64 and Intel, Windows x64, and an experimental Linux path without making Linux delay macOS or Windows. A platform-native API may not have the requested language installed, and an opaque OS service cannot be described as unloadable merely because DeskGraph drops its request objects.

## Decision

- Use a native-first provider stack: Apple Vision on macOS and `Windows.Media.Ocr` on Windows. Every native provider performs a runtime capability check for Traditional Chinese plus English before accepting work. Vision requires exact `zh-Hant` and `en-US`. Windows requests `zh-TW` and `en-US`, then validates the actual `RecognizerLanguage`: the first pass must resolve to a Traditional Chinese tag (`zh-Hant`, `zh-TW`, `zh-HK`, or `zh-MO`, never `zh-Hans`) and the second to `en-*`. Missing or semantically incompatible capability is `OcrLanguageUnavailable`, not a silent script downgrade.
- Use a separate `OcrProvider` interface. Providers receive core-validated bounded encoded bytes or a core-owned bounded raster plus dimensions, pixel format, orientation, deadline, and cancellation token. They never receive a path, URL, environment-controlled model directory, network client, process capability, or arbitrary file handle.
- The macOS adapter uses Vision's `VNRecognizeTextRequest` and `VNImageRequestHandler.initWithData:options:` through exact `objc2-vision 0.3.2`. Only the minimal Apple-framework features are enabled, and the dependency is target-specific. The provider checks supported languages at runtime and requests `zh-Hant` before `en-US`, because the verified mixed-language fixture requires that priority on the current runtime.
- The Windows adapter uses the already locked Microsoft `windows 0.61.3` binding. Its target-specific direct edge disables defaults and requests `std`, `Globalization`, `Graphics_Imaging`, `Media_Ocr`, `Storage_Streams`, and `Win32_System_WinRT`; Cargo still expands namespace parents and unifies features requested by other users of the same package/version. It writes owned bounded bytes to an `InMemoryRandomAccessStream`, decodes a `SoftwareBitmap`, initializes a dedicated MTA worker with balanced `RoInitialize`/`RoUninitialize`, and runs separate Traditional Chinese and English passes before bounded exact spatial de-duplication.
- Every WinRT async stage receives `Cancel()` on durable cancellation or deadline and calls `Close()` only after a terminal status. The caller returns within its deadline while an owned cleanup worker may continue draining native cancellation; a process-wide one-worker gate prevents stuck native work from accumulating and keeps OCR unavailable until cleanup finishes. If `Status()` itself fails, DeskGraph requests cancellation and releases the interface without falsely asserting a terminal state. Real Windows cancellation, cleanup convergence, CPU/RSS, and restart recovery remain acceptance gates.
- Windows publishes source-aligned boxes only when `OcrResult.TextAngle` is absent or zero. A non-zero or non-finite angle fails closed until a verified source-coordinate transform exists. `OcrWord` has no confidence API, so Windows stores `NULL`. The API is supported for desktop apps only with package identity, so this native path is conditional on MSIX/external-location identity and cannot replace the future packaged fallback for an unpackaged CLI or installer path.
- The cross-platform fallback is an in-process, bytes-only Tesseract adapter with packaged `eng` and `chi_tra` trained data. It may use neither the Tesseract CLI nor ambient data/dynamic-library discovery. Tesseract, Leptonica, binding, trained-data versions, hashes, multi-language initialization, packaging, cancellation, memory, notices, and all target builds remain separate acceptance gates; no fallback dependency is accepted by this ADR.
- PaddleOCR with ONNX Runtime is deferred to a later optional quality provider. It may be reconsidered only with DeskGraph's own screenshot corpus, macOS packaging evidence, telemetry-disabled runtime evidence, checksums, and unload/memory measurements. Pure-Rust `ocrs` is rejected for v0.1 because its current official model support is Latin-only and does not meet the Traditional Chinese requirement.
- OCR inputs are independently bounded below the generic image-metadata ceiling. Source bytes, dimensions, total pixels, output bytes, observation/chunk counts, and active processing time are checked before atomic publication. Cancellation and deadline requests must reach native work where the platform API permits; no partial native result is published after cancellation or timeout.
- Each stored OCR chunk records provider/version, source snapshot, one-based observation number, fragment index, normalized bounding box, optional provider-reported confidence, and `untrusted_extracted_text`. Bounds are always required. Confidence is stored only when the provider exposes a real score: Apple Vision supplies one, while `Windows.Media.Ocr.OcrWord` exposes only `Text` and `BoundingRect`, so Windows stores `NULL` rather than fabricating a number. An observation without valid bounded spatial provenance or with an invalid reported confidence is rejected instead of receiving fabricated byte offsets.
- Native APIs may cache OS models. DeskGraph may claim only that its own provider objects and buffers are dropped after native work reaches a terminal state; on Windows, caller return may precede a single draining cleanup worker. Release evidence must measure RSS before, during, after caller return, and after cleanup, and must not equate object drop with unloading an OS-owned model.

## Consequences

- macOS and Windows can use their maintained local platform OCR without shipping a model in the primary path, while the provider contract still admits a deterministic packaged fallback.
- The core remains functional when OCR is disabled or unavailable. OCR cannot become a hidden requirement for manifest scan, metadata/FTS search, Folder Profiles, or safe organization.
- macOS Vision runtime evidence and Windows provider code/cfg evidence do not close M2 or the cross-platform fallback requirement. Real Windows, Tesseract, representative corpus quality, Intel, clean-machine, 8 GB, installer, SBOM, and checksum evidence remain explicit gates.
- Apple Vision processing can fail in a restricted runner even when the runtime language probe succeeds. The 2026-07-17 local spike returned a nil platform error inside the restricted sandbox but recognized `DeskGraph OCR` and `桌面圖譜 安全整理` from the same controlled PNG through both Swift and Rust outside that sandbox. Runtime tests must expose, not hide, this environment boundary.
- `Windows.Media.Ocr` requires package identity and an installed recognizer to which each requested language can resolve. DeskGraph never installs a language capability silently. Missing identity returns `OcrProviderUnavailable`; missing or incompatible Traditional Chinese/English resolution returns `OcrLanguageUnavailable`. No automatic fallback routing exists yet; the separately packaged in-process provider remains a required future gate.

## Rejected alternatives

- Require Python, Docker, Ollama, a local LLM, or an external OCR service.
- Pass screenshot paths or URLs directly to native frameworks.
- Spawn Tesseract or another OCR CLI.
- Adopt one cross-platform model/runtime before native capability and resource measurements.
- Store OCR text with fake byte/page offsets or publish partial observations.
- Treat declared platform language support, object destruction, or an upstream benchmark as DeskGraph release evidence.

## Verification and revisit trigger

The dependency-selection spike compiled the published `objc2-vision 0.3.2` source with Rust 1.97 and only `VNObservation`, `VNRecognizeTextRequest`, `VNRequest`, `VNRequestHandler`, `VNTypes`, `block2`, `objc2-core-foundation`, and required Foundation features. A progress handler that reads an atomic cancellation flag and calls `VNRequest.cancel`, plus normalized bounding-box access, compiles; the mixed-language runtime probe returned both text observations with valid boxes. Runtime cancellation remains an implementation test, not accepted evidence. The complete nine-package isolated lock had no cached RustSec finding, and only `objc2-vision` is absent from DeskGraph's existing lock. The published crate archive SHA-256 was `bfc194758a2d5d7540b1ad283bfb9ca318ec608991892326e95b428230b2689b`.

The Windows binding gate inspected the published `windows 0.61.3` source and compiled its in-memory stream, decoder, resolved-language, OCR, `TextAngle`, bounding-box, MTA, async status, `Cancel()`, and terminal `Close()` APIs for `x86_64-pc-windows-msvc` with Rust 1.97. The production module plus platform-neutral state-machine/policy fixtures pass host tests and Windows cfg `cargo check`/Clippy when bundled SQLite is bypassed only for Rust typechecking. A normal macOS-to-MSVC check still stops in `libsqlite3-sys` because this host lacks Windows C headers; this is not link, executable, MSIX, or runtime evidence. Reusing the version already present through Tauri adds no lockfile package; its 9,372,520-byte crate archive has SHA-256 `9babd3a767a4c1aef6900409f85f5d53ce2544ccdfaa86dad48c91782c6d6893`, MSRV 1.74, and MIT OR Apache-2.0 license. The current 493-package workspace RustSec scan covers that closure with zero vulnerabilities and the documented unrelated 17 Tauri/Linux warnings.

Before any OCR provider is called complete, pass valid mixed Traditional Chinese/English, no-text, corrupt, extension/signature mismatch, oversized source/pixels/output, invalid provenance, cancellation, deadline, source-change, atomic replacement, migration preservation, FTS, and privacy-redaction fixtures. Providers that report confidence must pass its range validation; providers without a confidence API must persist explicit absence. Revisit this decision if either native API cannot consume bounded bytes/raster, cannot expose the required languages, cannot meet the release memory budget, or cannot provide a safe cancellation boundary.

The first macOS implementation slice now passes those behaviors at the core/provider/database boundary: bounded PNG/JPEG bytes, mixed-language and no-text output, corrupt/unsupported input, source/pixel/output/observation/chunk/deadline limits, spatial/confidence validation, durable cancellation through a blocking fake provider, source-change invalidation, operation-isolated atomic replacement, migration preservation, FTS and CLI redaction. A current-binary Apple Vision run outside the restricted sandbox recognized the controlled mixed fixture and published two spatial chunks; the restricted runner failed with a fixed provider error and zero output. This verifies the macOS arm64 slice locally, not actual runtime `VNRequest.cancel`, macOS Intel/Universal, Windows, packaged fallback, representative corpus, RSS/8 GB, clean-machine, installer or release acceptance.

The Windows implementation slice now covers package-identity preflight, owned bytes, separate requested language passes with resolved-capability validation, nullable confidence, source-aligned angle policy, strict rectangles, deterministic exact de-duplication, cancellation/terminal-close state transitions, bounded caller return, single-worker cleanup gating, and complete-only publication at the shared service/database boundary. These are code, host-policy-test, and Windows-cfg typecheck results only. They do not prove WinRT runtime behavior, package identity, installed recognizers, actual OCR quality, native cancellation/cleanup, RSS, installer integration, or fallback routing.
